s3断点下载

  1. 两个概念

    http request中的请求头有一个header为 range

    该字段表明请求的下载流从什么地方开始下载

    例如

    • RANGE： bytes=2000070-

    表明从2000070开始一直下载到最后

    • RANGE： bytes=2000070-2000080

      表明从2000070开始一直下载到2000080

    如果在请求时使用了range字段，那么在返回的response中会有多出一个header即

    Content-Range=bytes 100000-19999/20000

    表明你下载的返回是100000-19999，一共有20000字节，这个字段非常重要，服务器根据请求头的Content-Range来决定从文件流的哪个位置上开始读，并且需要返回状态码为206,表示部分请求成功，这些都是附加信息，告诉服务器需要怎样返回流

IO多路复用

发表于 2022-01-07 分类于 计算机网络 Valine：
本文字数： 1.6k 阅读时长 ≈ 1 分钟

整体思想

将被阻塞的操作比如accept()、read()、write()等操作交给操作系统内核进行处理，也就是我程序处理的一定是不会被阻塞的操作

注意：每个文件描述符都有两个缓冲区：写缓冲区和读缓冲区

Select（跨平台）

实现方式

  1. 将已连接的Socket都放入到一个文件描述符集合
  2. 调用select函数将文件描述符集合==拷贝到内核中==
  3. 内核遍历集合并检查是否有事件产生。如果有事件产生则将该Socket标记为可读或可写
  4. 将整个文件描述符集合==拷贝至用户态中==
  5. 用户态遍历找到可读或可写的socket并进行处理

函数

参数含义：

nfds：委托内核检测的三个集合中的最大的描述符+1

readfds：文件描述符集合 ，内核检测该集合中文件描述符对应的读缓冲区

writefds：文件描述符集合 ，内核检测该集合中文件描述符对应的写缓冲区

exceptfds：文件描述符集合 ，内核检测该集合中文件描述符是否有异常状态

timeout：超时时间 (如果为0，表示不会阻塞，立刻返回，如果不为0，则会持续监听一段时间再返回)

这三个文件描述符集合都是同时进行检查和操作的

readfds、writefds、exceptfds这三个集合的大小是和内核linux FD_SETSIZE参数定义的大小一致的

这三个集合既是==传入参数==也是==返回参数==，其中集合是以bitMap的形式存储的，我们这里以readfds为例：

  • 传入的bitMap

    如果传入的readfds中对应的bitMap中2，4，7这三个是需要检测的，传入的bitMap状态为00101001000...其中bitMap[2]、bitMap[4]、bitMap[7]都是1，其余位都是0,1的话表示==检测==这个文件描述符的读缓冲区，0表示==不检测==该文件描述符的读缓冲区

  • 传出的bitMap

    如果当前内核中2和4对应的文件描述符的缓冲区有数据，7对应的文件描述符没有数据，那么传入的bitMap中的2和4不变，表示该位对应的文件描述符的==读缓冲区有数据==，7的位置变为0，表示7这个位置的文件描述符的读缓冲区==没有数据==

这里对于文件描述符集合还有一些api操作函数

缺陷

  1. 发生了两次遍历和拷贝
  2. select使用固定长度的BitsMap表示文件描述符集合，但是所持有的文件描述符的个数是有限制的（因此每个进程默认持有的文件描述符是有限制的，默认为为1024个，但是select函数（系统自带的）默认中会使用linux FD_SETSIZE整个参数作为默认的最大文件描述符的个数，整个数在linux平台下是写死的，除非你重新编译内核对函数本身进行修改）

poll（Linux）

实现方式和Select差不多，最大的区别在于poll ==不再⽤ BitsMap 来存储所关注的⽂件描述符==，取⽽代之⽤==动态数组（该数组的个数用户可以自定义，甚至可以使用链表的方式）==，不受进程最大文件描述符的约束，但是依旧会受到==系统最大文件描述符个数的限制==

  1. TCP/IP四层模型

    应用层 负责向用户提供一组应用程序 http、DNS、FTP、telnet
    传输层 负责端到端的通信 tcp、udp
    网络层 服务网络包的封装、分片、路由、转发 ip、icmp
    网络接口层 负责网络包在物理网络中的传输，网络包的封帧、MAC寻址、差错检测、网卡传输网络帧

  2. 数据封装

    注意：因为物理链路层不能传输任意大小的数据包，因此有一个最大传输单元（MTU）为1500字节，也就是每次传送的IP包的最大大小，如果超过则需要在网络层进行分片

  1. 在每一层的网络包都是采用的相同结构体，从上往下就不断加头，从下往上就不断剪头（sk_buf->data传递给上一层）

    这样的好处在于在各层之间传递数据的时候不必要发生拷贝

创建TCP连接的步骤

准备阶段

服务端

  1. 指定网络协议（IPv4/IPv6）和传输协议（TCP的Socket）
  2. 绑定端口
  3. 绑定IP

​ TCP的组成：本机IP, 本机端⼝, 对端IP, 对端端⼝

客户端

明确服务端IP和端口号

连接阶段

  1. 客户端向服务端发送syn发起握手，客户端进入syn_sent状态
  2. 服务端接收到syn后进入syn_recv状态，此时内核将连接存储到半连接队列,并向客户端回复syn+ack
  3. 客户端收到syn+ack后，回复ack并进入established状态
  4. 服务端收到ack后，内核将连接从半连接队列中取出，并添加到全连接队列，服务端进入established状态
  5. 服务端应用进程调用accept函数，将连接从全连接队列中取出

内部细节

  1. 在连接过程中内核会维护两个队列

    • TCP半连接队列（未完成1）

    • TCP全连接队列（已完成1）

      具体参考https://network.51cto.com/art/202110/687595.htm

  2. 当TCP全连接队列不为空时，服务端的accept函数会从全连接队列中拿出一个已经完成连接的Socket返回应用程序

  3. 监听的Socket和传输数据的Socket是不同的两个Socket

    • 监听Socket是通过socket()创建的，然后bind()函数将其绑定到对应的地址和端口上，然后进程/线程就可以监听该socket
    • 传输数据的Socket是监听到TCP请求并三次握手之后通过accept()函数返回的socket（此处的socket是TCP全连接队列中的socket）

其他

  1. 理论上TCP最大连接数的数量

    最⼤ TCP 连接数 = 客户端 IP 数×客户端端⼝数 = （232*216）

    但是会收到文件描述符和系统内存的限制

  2. TCP连接数量受到的限制 （参考https://cloud.tencent.com/developer/article/1411022）

    • 单个进程最大文件描述符的限制

      默认是1024，可以使用ulimit -n进行修改

    • 内存的限制

      socket对应一个数据结构，会占用一定的内存

    • 全局限制

      cat /proc/sys/fs/file-nr会得到三个参数

      1.已经分配的文件句柄数，2.已经分配但没有使用的文件句柄数，3.最大文件句柄数

      可以修改最大文件句柄数，此处的3对应的是整个系统对应的最大文件句柄数

    • 客户端IP和客户端端口数的限制

      最⼤ TCP 连接数 = 客户端 IP 数×客户端端⼝数 = （232*216）

何为动态路由

我们都知道在go中使用如果想要访问的网址得到相应，那么每一个路由都应该对应一个handler对其进行处理

其中handler是一个处理/hello路由的函数

当我们去访问\hello时便会得到一个相应，那就是浏览器输出Hello test

但是这样带来了一种比较麻烦的情况，比如我现在有多个路由：

  • /hello/Cooper
  • /hello/Marry
  • /hello/Aaron

我们可以看到这里有三个路由，虽然不一样，但是基本形式上是完全一致的，我们可以归结为一下一种形式：

/hello/{name}

假设我们需要根据员工的name返回员工的信息，一般的做法就是定义一个路由，比如说是/hello/employeeInfo，然后通过Get方法传递员工的name给后端。

但是如果我们不使用传递参数的方法给后端呢？

按照常规的做法就是像上面定义三个路由，然后分别写对应的handler。

如果员工不止三个而是有非常多的人呢？显然我们不可能写那么多的路由以及对应的handler

这时候也许你就会想了，要是我们这里只写一个路由那该多好呀？比如说/hello/{name}，然后写一个handler去处理那就行了。

但是这里有一个问题：如何让程序知道我们请求形如/hello/{name}的路由是合法的呢？

这就需要动态路由来帮助我们处理了。

https证书搭建

发表于 2022-01-07 更新于 2021-08-27 分类于 cas Valine：
本文字数： 2.9k 阅读时长 ≈ 3 分钟

必须要安装java环境

  1. 生成服务端证书

    keytool -genkey -v -alias {证书名称} --keyalg RSA -keystore {地址/证书名称.keystore} --validity {有效期,单位为天}

    例如

    keytool -genkey -v -alias tomcat --keyalg RSA -keystore /root/https/tomcat.keystore --validity 36500

    箭头指向的地方代表了访问时候的ip地址，如果写localhost就必须用localhost访问，需要完全用红色箭头处的值来进行访问，ip就用ip访问，域名就用域名访问

  2. 生成客户端证书

    keytool -genkey -v -alias {证书名称} -keyalg RSA -storetype PKCS12 -keystore {地址/证书名.p12}

    例如

    keytool -genkey -v -alias mykey -keyalg RSA -storetype PKCS12 -keystore /root/https/mykey.p12

    客户端的CN可以是任意值

  3. 让服务端信任客户端证书

    由于是双向ssl认证，服务器必须要信任客户端证书，因此必须把客户端添加为服务器的信任认证。由于不能直接将PKCS12格式的证书库导入，必须先把客户端证书导出为一个单独的CER文件，使用如下命令：

    keytool -export -alias mykey -keystore {地址/客户端证书} -storetype PKCS12 --storepass {password} -rfc -file {地址/CER文件.cer}

    例如：

    keytool -export -alias mykey -keystore /root/https/mykey.p12 -storetype PKCS12 --storepass 123456 -rfc -file /root/https/mykey.cer

  4. 将该文件导入到服务器的证书库，添加为一个信任证书，命令如下

    keytool -import -v -file {地址/cer文件} -keystore {地址/服务器端证书}

    例如：

    keytool -import -v -file /root/https/mykey.cer -keystore /root/https/tomcat.keystore

  5. 查看服务器的证书库

    keytool -list -keystore {地址/服务器端证书}

    keytool -list -keystore /root/https/tomcat.keystore

  6. 让客户端信任服务器证书

    由于是双向ssl认证，客户端必须要信任服务端证书，因此必须把服务端添加为客户端的信任认证。由于不能直接将keystore格式的证书库导入，必须先把服务端证书导出为一个单独的CER文件，使用如下命令：

    keytool -keystore {地址/服务端证书} -export -alias {服务端证书名} -file {地址/服务端的证书名.cer}

    例如：

    keytool -keystore /root/https/tomcat.keystore -export -alias tomcat -file /root/https/tomcat.cer

  7. 最终一个会有4个文件

  8. 在tomcat的conf/server.xml中添加ssl配置

    属性说明：

    clientAuth：设置是否双向认证，默认为false

    keystoreFile：服务器证书文件路径

    keystorePass：服务器证书密码

    truststoreFile：用来验证客户端证书的根证书，此例子中就是服务器证书

    truststorePass：根证书密码

  9. 启动tomcat

    ./catalina.sh run
