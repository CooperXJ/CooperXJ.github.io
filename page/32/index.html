RDB文件的创建与载入

创建

种类

SAVE

  • 指令 (强制save)

    save

  • 该种方式会直接阻塞服务器进程，直到RDB文件创建完毕为止

  • 期间任何命令无法得到响应

BGSAVE

  • 指令

    bgsave

  • 该种方法会派生出一个子进程负责创建RDB文件，服务器进程继续处理命令请求

自动间隔性保存

配置

用户可以通过save选项设置多个保存条件，一旦其中某一个条件满足则会执行bgsave命令

比如

原理

  • 在redisServer结构中有属性

    • saveparams数组

      该数组保存了save选项设置的保存条件

      • seconds 秒数
      • changes 修改次数

    • dirty 修改计数器

      记录距离上次成功执行save命令或者bgsave命令之后服务器对数据库状态进行了多少次修改

    • lastsave 上一次执行保存的时间

  • 检查保存条件是否满足

    redis服务器周期性操作函数serverCron默认每个100ms就会执行一次，其中有一项工作就是检查save选项所设置的保存条件是否已经满足，如果满足则执行bgsave

载入

与AOF比较而言

  • AOF文件的更新频率比RDB文件高

  • 如果服务器开启了AOF持久化功能，那么服务器会优先使用AOF文件还原数据库状态

  • 只有在AOF持久化功能处于关闭状态时，服务器才会使用RDB文件来还原数据库状态

RDB文件结构

完整结构

REDIS db_version databases EOF check_sum

  • REDIS (5字节)

    开头标识符（就是redis这个字符串），表示载入的是RDB文件

  • db_version （4字节）

    RDB文件版本号

  • databases

    包含0个或任意多个数据库，以及各个数据库中的键值对数据

  • EOF （1字节）

    表示RDB文件正文内容结束

  • check_sum （8字节）

    校验和

databases

REDIS db_version database 0 database 3 EOF check_sum

该列表表示服务器的0号数据库和3号数据库非空

每个非空数据库的结构

SELECTDB db_number key_value_pairs

  • SELECTDB (1字节)

    表示读取数据库操作

  • db_number

    数据库号码

  • key_value_pairs

    保存数据库中的所有键值对数据

key_value_pairs结构

不带过期时间的键值对
TYPE key value

  • type

    代表一种对象类型或者底层编码

  • key

    键

  • value

    值

带过期时间的键值对
EXPIRETIME_MS ms TYPE key value

  • EXPIRETIME_MS

    告知读取程序接下来要读入的将是一个以ms为单位的过期时间

  • ms

    unix时间戳，键值对的过期时间

Value编码

字符串对象 （REDIS_RDB_TYPE_STRING）

  • REDIS_ENCODING_INT

    ENCODING integer

  • REDIS_ENCODING_RAW

    • 无压缩字符串（长度<=20字节）

      len string

    • 压缩字符串（长度>20字节）

      REDIS_RDB_ENC_LZF compressed_len origin_len compressed_string

      • REDIS_RDB_ENC_LZF

        压缩算法

列表对象（REDIS_RDB_TYPE_LIST）
list_length item1 item2 … itemN
集合对象 （REDIS_RDB_TYPE_SET）
set_size elem1 elem2 … elemN
哈希表对象 （REDIS_RDB_TYPE_HASH）
hash_size key_value_pair 1 key_value_pair 2 … key_value_pair N

每个键值对的结构

key1 value1 key2 value2 …
有序集合对象 （REDIS_RDB_TYPE_ZSET）
sorted_set_size element1 element2 … elementN

element结构

member score
INSET编码集合 （REDIS_RDB_TYPE_INTSET）

先将整数集合转换为字符串对象，然后将这个字符串对象保存到RDB文件中

ZIPLIST编码的列表、哈希表或者有序集合 （REDIS_RDB_TYPE_（LIST|HASH|ZSET)__ZIPLIST)

将压缩列表转换为一个字符串对象，然后保存到RDB文件中

分析RDB文件

  • 以ASCII码打开

    od -c xxx.rdb

  • 以16进制打开

    od -x xxx.rbd

Redis客户端的在服务器端的表现结构

Redis服务器结构中有一个属性为clients，是一个链表，每当有一个新的客户端与服务器进行连接时，会将该客户端添加了链表的末尾

客户端属性

套接字描述符 fd

  • -1

    伪客户端的命令请求来源于AOF文件或者Lua脚本，而不是网络

  • 大于-1的整数

    普通客户端

可以使用client list查看当前所有连接服务器的客户端的情况

客户端名字 name

可以使用client setname xxx给当前客户端取一个名字

标志 flag

含义

定义了客户端的角色（比如是master还是slave）以及当前客户端所处的状态

输入缓冲区 querybuf

  • 功能

    用于保存客户端发送的请求命令

  • 限制

    该缓冲区会根据输入内容动态地缩小或者扩容，但它的最大大小不能超过1GB，一旦超过会关闭该客户端

命令与命令参数

含义

服务器将客户端发送的命令请求保存到客户端状态的querybuf属性之后，服务器将对命令请求的内容进行分析，并将得出的命令参数以及命令参数的个数分别保存到客户端状态的argv属性和argc属性

argv

一个数组，数组中每一项都是一个字符串其中argv[0]为要执行的命令

argc

记录argv数组的长度

命令实现函数

服务器根据argv[0]在命令表中查找对应的实现函数

输出缓冲区

定义

执行命令所得的回复都会被保存在客户端状态的输出缓冲区中

输出缓冲区的种类

  • 固定大小的缓冲区（默认为16KB） buf

    存放长度较小的回复

  • 可变大小的缓冲区 reply

    存放长度比较大的回复

输出缓冲区的限制

  • 硬性限制

    输出缓冲区的大小一旦超过硬性限制那么该客户端就会被关闭

  • 软性限制 obuf_soft_limit_reached_time

    输出缓冲区介于软性限制和硬性限制之间的话，服务器会一直监视客户端，如果客户端的输出缓冲区大小在规定时间内一直处于这种状态，那么会将客户端关闭，如果在规定时间内不再超出软性限制，那么客户端就不会被关闭，同时对应的固定时间计时器会归0.

身份验证 authenticated

在开启身份验证的情况下

  • 0

    未通过

  • 1

    通过

时间

  • ctime

    客户端创建时间

  • lastinteraction

    最后一个客户端与服务器进行互动的时间

  • obuf_soft_limit_reached_time

    软性限制时间

Istio架构

特征

  • 连接
  • 安全
  • 策略
  • 观察

组件

  • Pilot

    服务发现和路由规则

    • Platform Adapter

      用来对接不同注册中心厂商，比如Eureka、etcd

    • Abstract model

      处理Platform Adapter对接的不同平台，从特定平台细节中解耦

    • Envoy API

      负责与Pod中的Envoy代理进行通信，主要包括服务发现还有流量控制规则转发给Envoy

  • Mixer （在Istio架构中不是必须的）

    控制策略，比如对服务调用限速

    Mixer模块

    • Policy

      用于向Envoy提供准入控制策略，黑白名单控制，速率限制等策略

    • Telemetry

      为Envoy提供数据上报和日志搜集服务，以用于监控告警和日志查询

    ​ Adapter适配的监控容器，比如普罗米修斯，就是搜集数据

  • Citadel （在Istio架构中不是必须的）

    管理各个服务的密钥和证书

    起到安全作用，可以保证服务之间的通信安全

  • Envoy

    代理，处理服务的流量

    • pilot-agent

      • 用于生成envoy配置

      • 负责启动envoy

      • 监控envoy

    • envoy进程的作用

      • 负责拦截pod流量
      • 从控制平面pilot获取配置和服务发现
      • 上报数据给控制平面组件mixer

  • galley

    相当于是微服务中的配置中心，给其他组件提供支持，统一管理配置文件

    • resource

      yaml配置文件

大概的流程：

  • 流量经过Envoy
  • 从Pilot中获取服务以及后端服务的负载均衡规则
  • 在Citadel中获取对应的密钥（安全相关的证书等等）
  • 向调用的服务发生请求
  • 将数据上报至Mixer，对流量进行监控

微服务的缺点

  • 代码侵入性太强
  • 微服务拆分越细，但是维护成本太高
  • 微服务是多语言调用，维护成本高
  • 微服务框架不断版本升级，会导致整个系统的代码可能出现问题 （比如之前API网关是使用zuul，不是SpringCloud gateway，而后者是使用webFlux技术则有些原先的业务代码很有可能出现问题）

设计思想

使用sidecar设计，将业务代码与通信完全分离
