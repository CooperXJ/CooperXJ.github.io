线程池的设计

最好每一类业务搞一个线程池

不要做全局的线程池，共用一个线程池，如果遇到某些任务处理数量大且时间较长，会影响到后续加到线程池任务队列里面的其他任务，尤其是一些对时效性要求较高的任务，并且不同类型的线程池作用也不相同

CPU密集型

最佳线程数 =1 + CPU核数

线程的数量一般会设置为“CPU 核数 +1”，这样的话，当线程因为偶尔的内存页失效或 其他原因导致阻塞时，这个额外的线程可以顶上，从而保证 CPU 的利用率。

synchronized 隐式锁

传统的synchronized在1.6版本之后性能已经和Lock差不多了，但是synchronized容易出现死锁

因为synchronized在申请资源的时候如果申请不到，线程会直接进入阻塞状态，而线程进入阻塞状态啥都干不了，也无法是否已经占有的资源，十分容易出现死锁的现象。

Lock

  1. 可以响应中断

    给阻塞的线程发送中断信号，阻塞的线程可以进行响应，释放曾经持有的锁，也就破坏了不可抢占的条件

  2. 支持超时

    如果一个线程在一段时间内没有获取到锁，不进入阻塞状态而是返回一个错误，那么该线程也有机会释放曾经持有的锁，破坏不可抢占的条件

  3. 非阻塞的获取锁

    如果尝试获取锁失败，并不进入阻塞状态，而是直接返回，那么该线程也有机会释放曾经持有的锁，破坏不可抢占的条件

管程的定义

管程，指的是管理共享变量以及对共享变量的操作过程，让他们支持并发

翻译为 Java 领 域的语言，就是管理类的成员变量和成员方法，让这个类是线程安全的

MASE模型 （java本身实现的管程模型）

  • 互斥

    管程本身提供的enq()与deq()两个方法，相互互斥

    只允许一个线程对其中的共享变量进行操作

  • 同步

    每个条件变量都有自己对应的等待队列

    条件变量的主要作用就在于协调各个线程之间的工作顺序

​ 执行顺序：如果条件变量不满足，那么就会进入对应的条件变量等待队列进行等待，当条件满足时就会进入到入口等待队列进行等待

并发编程出错的原因

发表于 2022-01-07 更新于 2022-01-17 分类于 java并发 Valine：
本文字数： 947 阅读时长 ≈ 1 分钟

非模式

并发编程一般主要由于以下三个方面经常导致出现bug

可见性

由于每个CPU都有自己的缓存，因此当使用多线程时会使用多个CPU，如果这些线程都操作同一个变量，那么会导致每个CPU对变量进行操作时其他CPU无法察觉到变化，那么也就无法取得数据的一致性。

原子性

因为大多数编程语言都是高级语言，每条语句都包含1~n条CPU指令，但是在进行多线程操作的时候无法原子性的控制一条语句的执行，最多只能控制到CPU指令的原子性执行，因此会导致一条语句有可能在其组成CPU指令的任意一条中断开转而去执行其他语句。

因此我们把一个或者多个操作在 CPU 执行的过程中不被中断的特性称为原子性

原子性的本质在于操作的中间状态对外不可见，解决原子性问题就是保证中间状态对外不可见。

有序性

大多数编译器为了优化性能可能会选择对原来的CPU指令的执行顺序进行改变，此时就有可能出现bug。比如利用双重检查创建单例对象，代码如下：

new操作应该是

  1. 分配一块内存
  2. 在内存上初始化singleton对象
  3. 将内存的地址赋给instance变量

但是实际上优化后的操作时：

  1. 分配内存
  2. 将内存地址分配给instance变量

信号量模型

组成

  • 一个计数器

  • 一个等待队列

  • 三个方法（这三个方法都是原子性的）

    • init()：设置计数器的初始值
    • down()：计数器的值减 1；如果此时计数器的值小于 0，则当前线程将被阻塞，否则当前线程 可以继续执行
    • up()：计数器的值加 1；如果此时计数器的值小于或者等于 0，则唤醒等待队列中的一个线 程，并将其从等待队列中移除

利用信号量来实现互斥锁

实现互斥的原理

当某一个线程执行到addOne()时，执行s.acquire();则当前的信号量的计数器为0，可以往下执行

当另一个线程执行到addOne()时，执行s.acquire();则当前信号量的计数器为-1，发生阻塞，进入信号量的等待队列

当第一个获取到执行权的线程执行完毕之后，执行 s.release();，则当前信号量变为0，则等待队列中的线程可以往下执行

读写锁 ReadWriteLock

互斥关系

读写锁允许多个线程同时读共享变量，但是写操作只能有一个线程操作

读锁与读锁之间不相互排斥，读锁与写锁相互排斥，写锁相互排斥

升级与降级

无法在持有读锁的情况下对锁进行升级为写锁

可以在持有写锁的情况下对锁降级为读锁

条件变量与公平模式

写锁支持条件变量，但是读锁不支持

读写锁类似于 ReentrantLock，也支持公平模式和非公平模式

性能较好的“读写锁” StampedLock

支持的锁类型

  • 写锁

    读写锁中的写锁

  • 悲观读锁

    读写锁中的读锁

  • 乐观读 （性能提升的主要原因）

    不加锁

乐观锁如何使用？

每次使用乐观读的读完之后，验证一下是否期间存在写操作，如果存在写操作，则将乐观读升级为悲观读锁

乐观锁验证读期间是否存在写操作原理

类似于在StampedLock中增加了一个version字段（也就是对应的stamp），每次写操作之后都会将对应的version字段+1，如果发现version前后不一致，则说明存在写操作

锁的升级与降级

stampedLock支持锁的升级tryConvertToWriteLock()与降级tryConvertToReadLock()

使用stampedLock的注意事项

  • 该锁不支持重入

  • 如果线程阻塞在 StampedLock 的 readLock() 或者 writeLock() 上时，此时调用该阻塞线程的 interrupt() 方法，会导致 CPU 飙升

    使用 StampedLock 一定不要调用中断操作，如果需要支持中断功能，一定使用可中断的 悲观读锁 readLockInterruptibly() 和写锁 writeLockInterruptibly()

Future

  • 作用

    获取线程运行的结果

  • 出现的场景

    线程池提交任务时，使用submit进行提交

  • 方法

    • 提交Runnable任务 Future<?>

      submit(Runnable task);

      runnable接口的方法是==没有返回值==的，所以 submit(Runnable task) 这个方法返回的Future仅可以用来断言任务已经结束了，类似于Thread.join()

    • 提交Callable任务 <T> Future<T>

      submit(Callable<T> task);

      这个方法是有返回值的，所以这个方法返回的Future对象可以通过调用其get()方法来获取任务的执行结果。

    • 提交Runnable任务及结果引⽤ <T> Future<T>

      submit(Runnable task, T result);

      假设这 个方法返回的Future对象是f，f.get()的返回值就是传给submit()方法的参数result。也就是会==返回传入的参数result==

      我们可以传入的参数result认为这是一个主子线程之间共享数据的桥梁

FutureTask

  • 作用

    是一个工具类

    FutureTask实现了Runnable和Future接口

    • 实现了Runnable接口

      所以可以将FutureTask对象作为任务提交给ThreadPoolExecutor去执行，也可以直接被Thread执行

    • 实现了Future接口

      所以能用来获得任务的执行结果
