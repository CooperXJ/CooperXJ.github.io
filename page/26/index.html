概念和存在意义

  • 概念
    • 最小部署单元
    • 一组容器的集合
    • 共享网络
    • 生命周期是短暂的
  • 存在的意义
    • 创建容器使用docker，一个docker对应一个容器，一个容器有一个进程，一个容器运行一个应用程序
    • Pod是多进程设置，运行多个应用程序，一个pod可以有多个容器，一个容器里面运行一个应用程序
    • Pod存在为了亲密性应用
      • 方便两个应用之间方便交互
      • 网络之间调用方法，可以不使用ip调用，直接使用socket调用或者127.0.0.1
      • 两个应用需要频繁调用（比如数据库的增删改查）

Pod共享网络与共享存储

  • 共享网络

    通过Pause容器，将其他业务容器加入到Pause容器里面，让所有业务容器在同一个名称空间中，可以实现网络共享

  • 共享存储

    引入数据卷概念Volume，使用数据卷进行持久化存储

Pod资源限制和要求

注：资源的限制是由docker做到的，而不是由pod做到的

在没有设置限制的时候，Pod会将宿主机的资源占用完

如果Pod内存不够，但是同时也其本身Pod的内存被限制那么会导致OOMKilled

建议

  • requests和limits资源定义推荐不超过1:2，避免分配过多资源而出现资源争抢，发生 OOM;

  • pod中默认没有定义resource，推荐给namespace定义一个limitrange，确保pod能分到资源;

  • 防止node上资源过度而出现机器hang住或者0OM，建议node上设置保留和驱逐资源， 如

    • 保留资源–system-reserved=cpu=200m,memory=1Gi

    • 驱逐条件–eviction hard=memory.available<500Mi

      是在/etc/sysconfig/kubelet这个文件中进行定义这些参数

Pod重启策略

  • Always （默认策略）

    不管是正常退出还是非正常退出都会重启

  • OnFailure

    非正常退出会进行重启

  • Never

    不会重启

每次重启之后都会有一个延迟，以指数后退延迟（10秒、20秒、40秒…）重新启动，以5分钟为上限，并在成功执行10分钟后重置。

Pod健康状态检测

因为有时候仅仅看pod的状态是无法判断当前pod是否可以接受服务的，比如java的堆内存溢出，Pod依然还是Running状态，但是实际上该应用服务已经不可以使用了，因此需要进一步进行检查

Pod调度

​ Pod调度主要会受到以下几方面的影响

  1. Pod资源限制和要求对Pod调用会产生影响

  2. 节点选择器标签影响Pod调度

    kubectl label node {node name} {label_name} = {label_value}

  3. 收到节点亲和性影响（和selector差不多，支持多个value选择）

  4. 污点和污点容忍

    • Taint污点：节点不做普通分配调度，是节点属性

    • 场景

      • 专用节点
      • 配置特定硬件节点
      • 基于Taint驱逐

    • 污点值有三种

      • NoSchedule：一定不被调度
      • PreferNoSchedule：尽量不被调度
      • NoExecute：不会调度，并且还会驱逐Node已有的Pod

    • 污点操作

      • 查看节点污点情况

        kubectl describe node {node name}|grep Taint

      • 为节点添加污点

        kubectl taint node [node] key = value {value为污点三个值}

      • 删除污点

        kubectl taint node [node] key = value {value为污点三个值+“-”}

    • 污点容忍

      指的是即使当前节点有污点，也会有一定可能性被调度到

QoS

定义

QoS主要用于Pod调度和驱逐时参考的重要因素，不同的QoS其服务质量不同，对应不同的优先级

如果Pod的优先级比较低，是有可能被驱逐的

k8s中的Pod在宿主机中是有优先级的

一共有三种优先级

  • Burstable （优先级第二）

    只要在资源中resource设置了requests和limits的时候(request<limits)，k8s会将其默认设置为Brustable类型的资源，k8s会保证其分配到的资源会在requests和limits中波动

  • Guaranteed（优先级第一（最高））

    只要在资源中resource设置了requests和limits的时候(request==limits)，k8s会将其默认设置为Brustable类型的资源，k8s会保证其分配到的资源为requests

  • BestEffort

    没有在资源yaml文件中定义resources，k8s会尽自己的最大努力分配资源给他，但是一旦该Pod影响到了其他优先级比他高的Pod，那么调度器会将该Pod进行驱逐

注意：

如果一个Pod中有多个容器，那么该容器的优先级是根据该Pod中的容器的最高等级进行制定的，比如有一个Pod中有2个容器，一个容器优先级为Burstable，一个为BestEffort，那么该Pod的优先级为Burstable

Service的作用

  1. 防止Pod失联，也就是服务发现

    因为对于pod来说它的ip总是不断变化的，因此如果一个服务想要访问一个pod，那么根据pod的ip是无法做到的，因此需要使用servcie进行服务发现

  2. 定义一组Pod访问策略（负载均衡,也就是VIP的作用）

    Service也可以起到负载均衡的作用，选择调度服务向多个同一种Pod中的一个进行发送请求

Pod和Service的关系

Pod和Service之间是通过标签建立联系的，Service自己也有Ip，但是它的Ip是VIp，也就是虚拟Ip，是会漂移的

Service的四种类型

  1. ClusterIP：集群内部使用

  2. NodePort：对外访问应用使用

  3. LoadBalancer：对外访问应用使用，公有云 (好处：主要是为了防止k8s中的node经常添加或者删除，它也是一层lvs，是根据NodePort做的lvs)

  4. ExternalName: 访问外部环境使用的，比如说外部有一个数据库（不在当前k8s集群中），可以通过ExternalName进行访问

注意：node内外部署应用，外网一般不能访问到，解决方法如下：

  • 使用NodePort，找到一台可以进行外网访问的机器，安装nginx，反向代理，手动将可以访问节点添加到nginx中
  • 使用LoadBalancer，利用公有云

    • 如果logs没有保存，则使用

      kubectl describe apiservice v1beta1.metrics.k8s.io

      查看是否报错

      一般出现的错误如下：

      FailedDiscoveryCheck

  3. 如果出现FailedDiscoveryCheck错误

    1. 检查是否可以ping通对应svc的ip

      如果无法ping通，则

      1. kubectl edit cm kube-proxy -n kube-system

      2. 修改mode为 mode: “ipvs”

      3. 添加ipvs的模块

      4. ```shell
        chmod 755 /etc/sysconfig/modules/ipvs.modules && bash /etc/sysconfig/modules/ipvs.modules && lsmod | grep -e ip_vs -e nf_conntrack_ipv4

      5. 再次尝试，应该可以ping通了

      详情见：https://blog.csdn.net/qq_29274865/article/details/108454912

    2. 检查是否可以执行kubectl top nodes

    3. 如果还是不行

      1. 检查 API Server 是否开启了 Aggregator Routing：查看 API Server 是否具有 –enable-aggregator-routing=true 选项

        ps -ef | grep apiserver* |grep enable-aggregator-routing

        如果没有显示则说明没有开启

      2. 修改每个 API Server 的 kube-apiserver.yaml(一般就是master节点下面有这个文件) 配置开启 Aggregator Routing：修改 manifests 配置后 API Server 会自动重启生效

        在command下添加- --enable-aggregator-routing=true

    1. 完成，执行kubectl top nodes

作用

在集群上管理和运行容器的对象

Pod和Controller的关系

Pod是通过Controller实现应用的运维。比如伸缩、滚动升级等等

Pod和Controller之间通过label标签建立关系

Deployment

本质上是replicas的升级版本，他是在replicas的基础上做的

Deployment应用场景

  • 部署无状态应用（web服务，微服务）
  • 管理Pod和ReplicateSet
  • 部署，滚动升级等功能

Deployment发布过程

  1. 导出yaml文件

    kubectl create deployment {name} --image={image name} -o ymal --dry-run > yaml文件地址

  2. 使用yaml文件部署应用

    kubectl apply -f {yaml文件}

  3. 对外发布（暴露对外端口号）

    kubectl expose deployment {deployment name} --port={应用内部自己的端口号} --type=NodePort --targte-port={对外访问目标端口号} --name={name} -o yaml > yaml文件地址

    kubectl apply -f {yaml文件}

应用升级、回滚和弹性伸缩

  • 升级

    • 应用升级

      kubectl set image deployment {name} nginx = nginx:1.15

    • 查看升级版本

      kubectl rollout history deployment {name}

    • 查看升级状态

      kubectl rollout status depolyment {name}

  • 回滚

    • 回滚到上一个版本

      kubectk rollout undo deployment {name}

    • 回滚到指定的版本

      kubectk rollout undo deployment {name} --to-version = {version 编号}

  • 弹性伸缩

    kubectl scale deployment {name} --replicas={数量}

DaemonSet 简介

DaemonSet 确保全部（或者一些）Node 上运行一个 Pod 的副本。当有 Node 加入集群时，也会为他们新增一个 Pod 。当有 Node 从集群移除时，这些 Pod 也会被回收。删除 DaemonSet 将会删除它创建的所有 Pod。

使用 DaemonSet 的一些典型用法：

  • 运行集群存储 daemon，例如在每个 Node 上运行 glusterd、ceph。
  • 在每个 Node 上运行日志收集 daemon，例如fluentd、logstash。
  • 在每个 Node 上运行监控 daemon，例如 Prometheus Node Exporter、collectd、Datadog 代理、New Relic 代理，或 Ganglia gmond。

一个简单的用法是，在所有的 Node 上都存在一个 DaemonSet，将被作为每种类型的 daemon 使用。 一个稍微复杂的用法可能是，对单独的每种类型的 daemon 使用多个 DaemonSet，但具有不同的标志，和/或对不同硬件类型具有不同的内存、CPU要求。
